## Overview

Tool governance has **two independent axes**:

- **Approval** — *does the agent need sign-off to run a tool?* Controlled by the agent's **Autonomy** setting.
- **Isolation** — *what can a tool actually reach?* Controlled by the sandbox tier, network egress, connector scope, and Deny rules.

Turning off approval does **not** turn off isolation. That separation is what makes "let the agent run on its own" safe: even a fully autonomous agent is still boxed in by its sandbox and can never touch anything a Deny rule forbids.

You set both on an agent's **Permissions** page.

## Autonomy (the approval axis)

Pick one mode per agent — the single control most agents need:

| Mode | Interactive chat | Headless routines | Use it when |
|------|------------------|-------------------|-------------|
| **Autonomous** *(default)* | Runs tools without an approval card | Runs tools to completion | You want the agent to just get the work done — especially routines, where there's nobody to approve. |
| **Supervised** | Shows an approval card for tools that require it | Skips (deny-fasts) tools that would need approval | You want a human to approve sensitive tools before they run. |

In both modes, a **Deny** rule always blocks a tool, and the sandbox isolation below is always enforced.

<Note>
  Autonomous is the default. A routine on an autonomous agent runs an `sandbox_exec` or research step to completion instead of silently failing for lack of an approver.
</Note>

## Isolation (always enforced)

These are the real guardrails — they apply in every Autonomy mode:

- **Sandbox tier** — code and shell tools run inside an ephemeral, isolated capsule. A rogue command only affects that throwaway sandbox.
- **Network egress** — off / allowlist / full. Off or allowlist means an autonomous agent can't reach the open internet or exfiltrate data.
- **Connector scope** — an agent only sees the connectors (and their secrets) you grant it.
- **Deny rules** — a hard ceiling. A tool matched by a Deny rule never runs, in any mode.

<Warning>
  Auto-approval is cheap for in-sandbox actions but not for **external, irreversible** connector actions (deleting a real Slack channel, sending a real email). For those, add a **Deny** rule, and keep network egress off or on an allowlist.
</Warning>

## Advanced: fine-grained tool policy

Under **Advanced** on the Permissions page you can go beyond the single Autonomy control:

- A **capability matrix** to set Allow / Ask / Deny per tool category.
- **Glob rules** to Deny (or Allow) specific tools or argument patterns.

Most agents never need this — the Autonomy control plus isolation covers the common case. Advanced is there for the ceiling you want to pin exactly.
